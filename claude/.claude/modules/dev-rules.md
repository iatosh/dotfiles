# CLAUDE.md

## 最優先で従うべき開発手法

### 基本開発プロセス

- **テスト駆動開発**: t-wada（和田卓人）が推奨する TDD の実践手法に従って開発する。ただし、RGR サイクルは Linus Torvalds の Atomic Commits を追加した RGRC サイクルに拡張する
  - **Red**: 失敗するテストを書く
  - **Green**: テストを通す最小限の実装をする
  - **Refactor**: コードを改善する
  - **Commit**: 動作する状態を原子的にコミットする
- **コード品質**: Robert C. Martin (Uncle Bob)が実践する Clean Code（SOLID 原則を含む）の手法で品質を確保する

### 基本設計原則

- **重複の排除**: Andy Hunt & Dave Thomas が推奨する DRY の実践方法を適用する
- **シンプルさの追求**: Kelly Johnson が実践した KISS の考え方を適用する
- **必要十分な実装**: Ron Jeffries が推奨する YAGNI の判断基準を適用する

### バージョン管理の実践

- **コミットの粒度**: Linus Torvalds が推奨する Atomic Commits の原則に従い、小さく完結した単位でコミットする
- **コミットメッセージ**: Chris Beams の 7 つのルールと Conventional Commits の形式を採用する
- **安全な変更**: 大きな変更や実験的な変更の前には必ず現在の状態をコミットする
- **ブランチ戦略**: 新機能は feature ブランチで開発し、動作確認後にマージする

### 要件と仕様の管理

- **要件の記述**: Mike Cohn が推奨する User Story 形式で記述し、INVEST の原則を満たすようにする
- **仕様の詳細化**: Gojko Adzic が実践する Specification by Example の手法で、具体的なシナリオを文書化する
- **文書の維持**: Cyrille Martraire の Living Documentation の考え方に従い、文書とコードを常に同期させる
- **文書の保存先**: `REQUIREMENTS.md`に一元管理する

### 開発記録の実践

- **開発日誌**: Andy Hunt が推奨する Developer Journal の手法で、進捗と学びを`DEVLOG.md`に記録する
- **記録の原則**: 技術的な懸念、コードの問題点、改善提案などを率直に記録する
- **記録のタイミング**:
  - 新機能の実装完了時
  - 重要な問題の解決時
  - 大きなリファクタリング実施時
  - 設計上の懸念が生じた時

## プロジェクトに応じて適用する手法

### Martin Fowler の実践手法

- **リファクタリング**: 彼のカタログに基づいた安全な改善手法を使用する
- **継続的統合**: 彼が提唱する自動化とフィードバックループを実現する
- **Feature Toggle**: 彼が推奨する段階的リリース手法を実装する

### アーキテクチャ設計

- **依存関係の整理**: Robert C. Martin の Clean Architecture 実践手法を適用する
- **API 設計**: Roy Fielding の REST 実装手法に従う
- **パターンの適用**: Gang of Four が示した実装例に基づいて選択する

### 構成と運用

- **テスト戦略**: Mike Cohn が実践する Test Pyramid の構成方法を採用する
- **アプリ構成**: Adam Wiggins が実証した Twelve-Factor 手法を実装する
- **インフラ管理**: Kief Morris が推奨する IaC の実践パターンを使用する

### 機械学習プロジェクトの設定管理（該当時）

- **実験の再現性**: Google Research (D. Sculley et al.) が提唱する MLOps の実践手法に従い、全パラメータを YAML で外部管理する
- **構造化設定**: Omry Yadan (Hydra) が実践する階層的な設定管理手法を適用する

### プログラミングパラダイム（該当時）

- **OOP**: Alan Kay が意図した本来のオブジェクト指向手法を実践する
- **FP**: Rich Hickey が実演する関数型プログラミング手法を適用する

## ドキュメントフォーマット

### REQUIREMENTS.md

```markdown
# プロジェクト要件定義

## 機能要件

### [機能名]

**User Story**
As a [具体的なユーザー役割],
I want [達成したい目標],
so that [ビジネス価値].

**Acceptance Criteria**
Given [前提条件]
When [アクション]
Then [期待結果]

**Examples**
| 状況 | 入力 | 期待される振る舞い |
|------|------|-------------------|
| 通常ケース | xxx | yyy |
| エッジケース | xxx | yyy |

**実装状況**: [ ] 未着手 / [x] 完了
```

### DEVLOG.md

```markdown
## エントリー [通し番号]

### 実装内容

- 何を実装したか

### 技術的な懸念事項

- コードの問題点や改善すべき箇所
- より良い実装方法の提案
- 現在の設計の限界や課題

### 所感

- 実装中に感じた技術的な課題
- もっとこうすべきだったという反省点
- このプロジェクトの技術的な方向性への提案

### 次回への申し送り

- 未解決の課題
- 検討が必要な事項
```

注：開発日誌は純粋に技術的な振り返りと改善のために使用される。遠慮なく問題点や改善案を記録すること。
