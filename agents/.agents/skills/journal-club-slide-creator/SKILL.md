# 雑誌会スライド作成スキル

論文を雑誌会・輪講向けのスライドに落とし込むための指針。
論文タイプに依存しない普遍的な原則と、TTLab固有のルールを収録する。

論文タイプ別の重点調整は `references/` 内の対応ファイルを参照すること：
- 解析系（EEG・神経科学・行動実験など） → `references/analysis-papers.md`
- 臨床AI・診断精度研究（深層学習モデル開発・専門家比較・外部検証など） → `references/clinical-ai-papers.md`
- （今後追加予定）

---

## 0. 作業の 2 段階構造

スライド作成は **「読解フェーズ」→「スライド化フェーズ」** の順で行う。
論文の章立てに引きずられたままスライドを作り始めると、聴衆視点の構成にならない。

---

## 0-A. 読解フェーズ（スライド作成前の必須工程）

### 読む順序

論文の章立てに従わなくてよい。以下の順が効率的：

1. **Abstract** → コアクレームと数値を把握する。「何を達成したか」を 1 文で言えるようにする
2. **Introduction の最終段落** → 著者自身が述べる貢献リスト。これが先行研究スライドの「本研究」行になる
3. **Figure / Table をすべてスキャン** → 論文全体の構造を視覚的に把握する（後述）
4. **Methods** → パイプラインの各ステップと設計上の工夫を把握する
5. **Results 本文** → 図に出ていない数値・サブグループ分析を拾う
6. **Discussion** → 著者の解釈と、著者が「言えていないこと」を区別する
7. **Supplementary** → 本文に根拠として引用されている手順・パラメータを確認する

### Figure / Table の読み方

図を読むとき、「何が描いてあるか」だけでなく以下を確認する：

- **フロー図（データパイプライン・アーキテクチャ図）** → スライドのフローチャートの骨格になる。ステップ数・分岐・フェーズの区切りを把握する
- **結果図（ROC・PR 曲線・棒グラフなど）** → 比較対象が何か、自モデルがどの位置にいるかを確認する。数値は本文から拾う
- **Table** → 参加者属性・データセット内訳・性能比較は表のままスライドに使えることが多い

### 先行研究との「ギャップストーリー」の把握

背景スライドの核心は「先行研究の限界 → 本研究の必要性」の流れ。
Introduction を読むとき、以下を明示的に探す：

- 先行研究 A は何ができて、何ができなかったか
- 先行研究 B は何ができて、何ができなかったか
- それら両方の限界を同時に解決するのが本研究、という論理になっているか

例：SpikeNet1 はイベントレベルに強いが偽陽性率が高く、SCORE-AI は EEG レベルに強いがイベントレベルに非対応。SpikeNet2 は両方を同時に解決する、という構造。

### 手法の「核心的な工夫」を特定する

Methods には多くの要素が含まれるが、スライドで最も丁寧に説明すべき工夫は 1〜2 個に絞る。
「この工夫がなければ本論文の主クレームは成立しない」という部分を探す。

例：SpikeNet2 における Hard-Negative Mining は偽陽性削減という主クレームの根拠であり、省略できない。

### 批判的視点の形成

論文を読み終えた時点で、発表者自身の疑問を 3〜5 個メモする：

- 著者が「成功」と言っている箇所で「本当にそう言えるか」と感じた点
- データ選定・評価設計に感じた違和感
- 論文の知見が自分の研究に当てはまるか・当てはまらないか
- 次に読むべき論文として何が浮かんだか

## 0-B. スライド化フェーズの開始前確認

- 読解フェーズの出力が揃っているか
- 発表時間の目安（例：20分、30分）
- 聴衆の専門知識レベル（同じラボでも B3 が聞いてわかるレベルを目指す）
- 発表形式（口頭のみ / デモあり / 質疑込み）
- **論文のタイプを確認し、該当する `references/` ファイルを読む**

---

## 1. スライド全体の構成テンプレート

```
1. タイトル・著者・発表者名・日付
2. 概要（全体を1スライドで）
3. 背景（問題設定・先行研究・課題）
4. 目的（※仮説と重複する場合は仮説を省略してよい）
5. 方法
6. 結果
7. 考察
8. 限界と今後の課題
9. 結論
10. コメント（発表者の批判的視点）
[オプション] Appendix（補足図・詳細手順・QA想定）
```

**命名の注意：**
- 最後のスライドは「まとめ」ではなく「**結論**」
- 「展望」ではなく「**今後の課題**」
- 目的を調べることが仮説なので、目的と仮説の両方を書く必要はない

枚数の目安：全体で 20〜30 枚（20 分発表の場合）。

---

## 2. タイトルと 1 スライド 1 キーポイントの原則

- スライドのタイトル＝「そのスライドで一番言いたいこと」
- タイトルだけで内容が推察できるのが理想
- タイトルで書いたことを本文 1 文目で繰り返さない
- 1 枚のスライドには 1 つのメッセージだけ
- キーポイントは上に書き、補足は下に書く
- どのスライドから見ても内容が伝わる構成（途中から聞いた人にも理解できること）

---

## 3. 情報の配置とレイアウト

**重要な情報は上に置く**
- スライド下部は前列の人の頭で隠れる可能性がある
- 図のキャプションをスライド最下部につけても見えない

**目線の移動を最小化する**
- 図の説明は図の中、または図の近くに置く

**箇条書きより図解化**
- 箇条書きは最小限にし、可能な限り図・フローチャート・表で表現する
- 1 項目だけの箇条書きは不要（文章で書く）
- 文末は体言止めか文章のどちらかに統一する
- 論文の文章をそのまま貼り付けない（口頭で説明すれば十分）

**矢印の使い方**
- 意味のない矢印は使わない
- 使う場合は明確な定義を持って使う
- 矢印を多用したくなる場合は構成の問題を疑う

---

## 4. 概要スライドの設計（最重要）

発表全体のロードマップ。聴衆がここで「何を聞けばよいか」を判断する。

**必須要素**
```
背景：  ～という問題があった（1〜2 行）
目的：  ～を明らかにすることを目的とした（1 行）
方法：  ～を用いて～を行った（1〜2 行）
結果：  ～が明らかになった（数値を含めて具体的に）
```

**やってよいこと**
- 右側に論文の代表図を 1 枚配置する（システム概要図・実験パラダイム図など）
- 結果は数値を含めて具体的に書く（「性能が向上した」ではなく「指標 X が Y% 改善した」）

**やってはいけないこと**
- 箇条書きを 5 行以上並べる
- アブストラクトをそのまま貼る
- 図のない文字だけのスライド

---

## 5. 背景スライドの設計

**情報の絞り込み順序（漏斗型）**

一般的なトピック → 本研究に関連する先行研究 → 先行研究の限界 → 本研究の必要性

**先行研究は比較表にまとめる**

論文に複数の先行研究が言及されている場合は表にまとめる。

| 研究 | 手法 | 主要指標 | 結果 | 備考 |
|------|------|----------|------|------|
| A et al. (20XX) | 手法α | 指標X | 値 | 課題 |
| 本研究 | 手法β | 指標X | 値 | 改善点 |

軸の選び方：本研究が改善した指標・本研究と直接対比できる軸を選ぶ。本研究の新規性が際立つ軸を必ず含める。

**先行研究と発表者自身の意見は同じスライドに混ぜない。**

**仮説の導き方**

仮説は先行研究の知見から自然に導けるものにする。「先行研究がないから調べる」は研究目的にならない。

---

## 6. 目的スライドの設計

```
目的：  ～を明らかにする
アプローチ：  解析対象・比較方法（手法の概要）
仮説（目的と重複しない場合のみ）：
        ～が生じると予測される
        ～が改善すると予測される
```

仮説はアプローチの後に置くことで、「どう検証するのか」が先に示され、仮説の論理的根拠が伝わる。

---

## 7. 方法スライドの設計

**フローチャートを積極的に使う**

実験手順・解析パイプラインは必ずフローチャートで表現する。

複数スライドに分割する場合：全体フローを毎スライドで表示し、当該ステップのみハイライトする。

```
[スライド N]    ①→②→③→④→⑤  （②がハイライト）
[スライド N+1]  ①→②→③→④→⑤  （③がハイライト）
```

**並列した複数の内容は構成を統一する**

複数の指標・手法を並べて説明する場合、全て同じ構成で記述する。

**Supplementary Material も確認する**

本文だけでなく補足資料の図・表も確認し、以下の場合は積極的に採用する：
- 本文の議論の根拠になっている詳細な手順
- 聴衆が「なぜそうするのか」を理解するために必要な情報

---

## 8. 結果スライドの設計

**論文の図には必ず注釈を追加する**

- 最も重要な数値や傾向を矢印・枠・テキストで強調する
- 「指標 X が Y% 向上」などの結論を図の横に添える
- 図のキャプションは図の近くに配置する

**結果の提示順序**

1. 主要な結果（本論文が最も主張したい発見）を最初に提示
2. 副次的な結果はその後
3. 先行研究との比較は考察スライドで行う

**検定結果の扱い**

- 統計検定の結果（p 値・効果量）は補助的な立ち位置
- 大きく載せる必要はなく、表の中に `*p<0.05` などの形で添える
- 検定結果だけのスライドは NG

**比較が必要な結果は表にする**

複数の条件・パラメータを比較する場合、棒グラフ（論文の図）と要約表の組み合わせが有効。

---

## 9. 考察スライドの設計

**考察の構造**

「個別の結果ごとの考察の羅列」ではなく、仮説に対する総合的な検証として書く。

```
仮説の再提示
  ↓
結果 1・結果 2・結果 3（まとめて提示）
  ↓
仮説の検証結果に対する考察（先行研究の知見で支持・反証）
```

**先行研究との定量的比較**

- 悪い例：「先行研究より高い性能を達成した」
- 良い例：「先行研究（A et al., 20XX）の指標 X = 値 に対し、本研究は 値 を達成した」

---

## 10. コメントスライド（必須）

発表の最後に必ず設ける。雑誌会のディスカッションの起点になる。

**批判的な指摘（優先度：高）**
- 方法論上の限界（論文の Limitations に書かれていても発表者自身の言葉で強調してよい）
- 一般化可能性への疑問（サンプル数・条件・データセットの制限など）
- 交絡変数の問題
- 比較対象・評価指標の妥当性への疑問

**ポジティブな評価**
- 研究の意義・インパクト、方法の巧みな点

**個人的な感想・発見**
- 印象的だった記述（英語論文なら原文 + 日本語訳を並置）
- 関連する自分の研究との接点
- 次に読みたい論文

論文の言葉をそのまま使わず、発表者自身の言葉で書く。

---

## 11. Appendix の活用

**QA スライド**

発表後の質疑応答を想定した問答を Appendix に用意する。

- 方法の詳細（「なぜその手法を選んだのか」）
- 先行研究との差異（「○○と何が違うのか」）
- 一般化可能性（「他の条件でも成立するか」）
- 限界（「この結果の解釈の限界は何か」）

## 12. TTLab 固有のルール

### NGワード・表現

| 使ってはいけない表現 | 代わりに使う表現・理由 |
|---|---|
| 従来 | 具体的な先行研究名で言及する |
| 最適 | 最適化と混同しやすい。より適切な語を選ぶ |
| 今後の展望 | 今後の課題 |
| イメージ | 曖昧なので使わない |
| 〜として | 共通認識前提になり読者に解釈を委ねる |
| 知見が乏しい | 具体的な研究の意義を示す（研究目的にならない） |

### 数値・単位の表記

- 文字に単位をつけるときは `[]` を使う：`t [s]`
- 数字に単位をつけるときは `[]` を使わない：`t = 0 s`（`t = 0 [s]` は NG）

### 引用の書き方

- `[Tanaka, et al., 2025]` 形式を機械的に使わず、文脈に応じた引用の仕方を選ぶ
- 日本語文献は日本語で引用（例：`[大平, 2025]`）
- 書籍の引用は論文と同様（書籍名 = 論文誌名として扱う）
- arXiv の論文も引用してよい

### 成功・失敗の判定

- 「成功した」「失敗した」は自分で決めない
- 結果を客観的に記述し、読者が判断できるようにする

## 13. 完成前チェックリスト

### 構成・命名
- [ ] 最後のスライドは「結論」か（「まとめ」「展望」ではないか）
- [ ] 「今後の課題」を「展望」と書いていないか
- [ ] 目的と仮説を不必要に両方書いていないか

### タイトルと情報配置
- [ ] 各スライドのタイトルで内容が推察できるか
- [ ] タイトルと本文 1 行目が同じ内容の繰り返しになっていないか
- [ ] 重要な情報がスライド上半分にあるか
- [ ] 図の説明が図の近くにあるか

### 先行研究・引用
- [ ] 先行研究と発表者自身の意見が同じスライドに混在していないか
- [ ] 先行研究は比較表にまとめているか
- [ ] 引用形式は文脈に応じているか（日本語文献は日本語表記か）

### 図・表・検定
- [ ] 論文の図に注釈・強調を加えているか
- [ ] 複雑な手順はフローチャートにしているか
- [ ] 検定結果だけのスライドを作っていないか
- [ ] 表にする意味・目的が明確か

### 文章・表現
- [ ] スライドに長い文章を貼り付けていないか
- [ ] 箇条書きが多すぎる部分を図解化できないか
- [ ] 1 項目だけの箇条書きを文章に書き直しているか
- [ ] 文末が体言止め／文章のどちらかに統一されているか

### TTLab NGワード
- [ ] 「従来」を使っていないか
- [ ] 「最適」を使っていないか
- [ ] 「今後の展望」を使っていないか
- [ ] 「イメージ」を使っていないか
- [ ] 「〜として」を不必要に使っていないか
- [ ] 「知見が乏しい」を研究目的にしていないか

### 数値・単位
- [ ] `t [s]`（文字）と `t = 0 s`（数字）の表記を区別しているか
- [ ] 数値（サンプル数・精度・p 値など）が論文と一致しているか

### 批判的読解
- [ ] コメントスライドに発表者自身の視点があるか
- [ ] 論文の限界を自分の言葉で説明できるか
- [ ] 仮説と結果の整合性を確認しているか
