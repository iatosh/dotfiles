# 臨床AI・診断精度研究の重点調整

深層学習モデルを開発し，臨床専門家と比較・外部検証する形式の論文に適用する。
EEGスパイク検出・画像診断・バイオマーカー検出など，「AIモデル vs 専門家」の構図を持つ論文が対象。

---

## このタイプの論文の構造的特徴

臨床AI論文は以下の 3 層構造を持つことが多い：

1. **モデル開発層**：アーキテクチャ設計・学習データ・訓練手法
2. **評価層**：内部テスト・外部検証・専門家比較
3. **臨床応用層**：偽陽性率・実運用シナリオ・限界

スライド構成はこの 3 層に対応させると聴衆が迷わない。

---

## 概念説明スライドを必ず設ける

### タスク定義の明示

「イベントレベル検出」と「EEGレベル分類」のように，同一論文内に複数のタスクが存在する場合，
それぞれのタスクを定義するスライドを背景セクションに置く。

```
タスク名：何を入力として，何を出力するか
          → 入出力を図で示す
臨床的意味：臨床現場でこのタスクがなぜ必要か
```

タスクを混同したまま結果スライドに進むと聴衆が混乱する。

### 評価指標の定義

臨床AI論文で頻出する指標は馴染みのない聴衆も多い。
以下の指標が登場する場合は，Appendix ではなく背景〜方法の間に定義スライドを置く：

| 指標 | 定義の要点 |
|------|-----------|
| AUROC | ROC 曲線の面積。感度 vs 特異度のトレードオフ全体を評価 |
| AUPRC | PR 曲線の面積。クラス不均衡があるとき AUROC より有用 |
| Calibration / Brier Score | 予測確率と実際の陽性率の一致度。臨床的信頼性の指標 |
| mAUC | FP/hour を横軸にした修正 ROC 曲線。連続モニタリング用 |

数式よりも「何を測っているか」「なぜこの指標が必要か」を優先して説明する。

## データ分割図は複数スライドで展開する

訓練・検証・テストの分割が複雑な場合（患者単位分割，フェーズ間の重複，外部データセットなど），
Figure 1 のような CONSORT 風フロー図を必ずスライドに取り込む。

フロー図を説明するスライド構成例：

```
[スライド A] 全体データソースの俯瞰
             → データセット名・患者数・施設数
[スライド B] 開発用データの内訳（訓練フェーズごとの分割）
[スライド C] テスト用データの内訳（内部・外部）
```

「同一患者が訓練とテストに重複しない」などのデータ汚染防止策は明示する。

---

## 専門家比較の設計を批判的に読む

「専門家レベルの精度を達成した」という主張は，比較設計によって意味が大きく変わる。
以下の点をスライドで整理し，コメントの材料にする：

**比較方法の違い**
- モデルの ROC 曲線に対して専門家の動作点（閾値固定）を比較 → モデルに有利
- 専門家も ROC 曲線を持つ場合（複数閾値）は対等な比較になる

**ラベルの扱い**
- 専門家の多数決をハードラベルとして使う場合，ラベルそのものを「正解」として学習したモデルとの比較は循環的になりうる
- ソフトラベル（投票割合をそのまま使う）を採用した場合との差異を検討する

**EOSN（モデルを上回る専門家の割合）の読み方**
- 44% の専門家がモデルを上回る = 56% がモデル以下 という解釈と，
- 44% という数字が信頼区間の幅に依存する点（例：25〜75% と幅広い場合は過信禁物）の両方を示す

---

## ハードネガティブマイニング（Hard-Negative Mining）

偽陽性削減を目的とした学習手法として頻出する。定義スライドを 1 枚設ける。

```
定義：モデルが誤って陽性と予測した「本来は陰性」のサンプルを
      繰り返し訓練データに追加することで，識別境界を精緻化する手法

直感的説明：「モデルが騙されやすいパターン」を集めて，
             そのパターンを陰性と認識するよう再訓練する

注意点：ラウンドを重ねすぎると過学習が生じうる（検証セットで確認が必要）
```

## 外部検証の位置づけ

内部テストセットと外部検証データセットで性能差がある場合，その差の要因を考察する：

- データ収集施設・EEG機器の違い
- 患者属性（年齢層・疾患の重症度・設定：外来 vs ICU）
- ラベリングプロトコルの差異

「外部検証で性能が下がった」だけでなく，「どのデータセットで，どの指標が，どの程度変化したか」を表にして示すと議論しやすい。

## 結果スライドの設計（臨床AI固有）

### 複数データセット × 複数指標の整理

内部テスト・外部検証 1・外部検証 2 で同じ指標を報告する場合，論文の図をそのまま貼るよりも
要約表を添えると聴衆の理解が早い：

| データセット | AUROC | AUPRC | Brier | mAUC |
|------------|-------|-------|-------|------|
| MGB（内部） | 0.973 | 0.995 | 0.02 | 0.997 |
| HEP（外部） | 0.942 | 0.948 | 0.065 | - |

数値だけでなく「どの数値が最も重要か・なぜか」を 1 行で添える。

### 偽陽性率（FP/hour）の提示

mAUC や FP/hour は臨床応用の文脈で意味を持つ。
「感度 90% 時点での FP/hour = 9 spikes/hour」のような具体的な動作点の数値を図に直接書き込む。

## 交絡変数への批判的視点（コメントスライド推奨）

臨床AI論文固有の批判的視点：

- **データ選定の偏り**：イベント候補をパターンマッチングで事前選択している場合，選択基準がモデルに有利な候補を生む可能性
- **ラベルの循環性**：専門家の多数決を正解ラベルとして使い，同じ専門家と比較することの論理的問題
- **施設バイアス**：単一の大病院に偏ったデータセットでは，他施設・他国への汎化性が保証されない
- **実運用とのギャップ**：「イベント単位」の評価指標が，実際の「連続モニタリング」運用でどの程度有効か
- **人口統計学的多様性の欠如**：アジア系・小児・ICU 患者など代表性が低いサブグループの性能保証が不十分な場合

## 参考：このタイプの論文で有効なスライド構成例

```
1. タイトル
2. 概要
3. 背景（臨床問題の説明 + 既存手法の限界）
4. タスク定義（イベントレベル / EEGレベルなど）
5. 評価指標の定義（AUROC・AUPRC・mAUC など）
6. 先行研究と課題（比較表）
7. 目的
8. データセット（患者属性・収集施設・分割方針）
9. データ分割フロー図（複数スライド）
10. モデルアーキテクチャ（概要図 + 設計上の工夫）
11. 学習の工夫（Hard-Negative Mining など）
12. 結果①：イベントレベル評価（内部・外部）
13. 結果②：EEGレベル評価（内部・外部・専門家比較）
14. 考察（外部検証での性能差・専門家比較の解釈）
15. 限界
16. 結論
17. コメント
[Appendix] データ分割詳細 / アーキテクチャ詳細 / 学習条件 / QA
```
