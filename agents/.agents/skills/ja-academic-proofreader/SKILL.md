# ja-academic-proofreader

田中研の **論文Tips / 日本語Tips** をルールブックとして、日本語の学術文を添削するスキル。
検出は2層構成:

- **静的層（スクリプト）**: `scripts/proofread.js` が textlint(prh) ＋ 形態素解析 ＋ 約物 ＋ 構造 ＋ LaTeX を
  検査し、ルールID付き JSON を返す。確実な誤り＝`error`、正用もありうるもの＝`warning`。
- **文脈層（あなた＝エージェント）**: 静的層の `warning` を文脈で精査し、機械では検出できない
  文脈依存ルール（H系・L系）を判定する。**外部LLMは呼ばない。あなた自身が判定する。**

全ルールの定義・✗/○の実例・検出方法は **`references/rules-catalog.md`** にある。本ファイルは手順と
判定の指針のみを置く。ルールIDの意味が不明なときは必ず catalog を参照すること。

---

## 手順

### Step 0. 入力と前提の確認

1. 対象を特定する。ファイルパスがあればそれを使う。ユーザーが本文を貼り付けた場合は
   一時ファイル（例 `/tmp/proofread-input.md`、LaTeXなら `.tex`）に保存してパスを得る。
2. 次の2点を簡潔に確認する（ユーザーの依頼から自明なら確認を省いてよい）。
   - **文書種別**: 予稿（学内・研究会）／学位論文・ジャーナル・国際会議論文／研究計画書／レポート。
     → 本稿・本論文・本研究の使い分け（H-11）と、序論で想定すべき読者（L-18）が変わる。
   - **文脈層の要否**: 既定は「実施する」。ユーザーが「textlintだけ」「静的だけ」と言った場合のみ
     静的層のみで止め、H系・L系は出力しない。

### Step 1. 静的層を実行

初回のみ依存をインストール:

```bash
cd ~/.agents/skills/ja-academic-proofreader && npm install
```

検査を実行（`<file>` は対象パス。種別は拡張子と内容から自動判別、`--type prose|latex` で上書き可）:

```bash
node ~/.agents/skills/ja-academic-proofreader/scripts/proofread.js <file> --format json
```

返る JSON は `{ inputType, summary, findings: [{ ruleId, severity, line, column, message, excerpt }] }`。
`severity` は `error`（S-hard）/ `warning`（S-soft）。これを文脈層の入力にする。

> 補足: ルールブックそのものを引きたいときは `npx textlint -c .textlintrc.json <file>` でも
> フレーズ系（prh辞書）だけは確認できる。正式な統合結果は常に `proofread.js`。

### Step 2. 文脈層（文脈層が必要なときのみ）

`references/rules-catalog.md` を読み、対象本文を通読したうえで次を行う。

**(a) S-soft（warning）の精査** — 静的層の warning は「候補」。文脈で正用/誤用を判定し、正用は落とす。
代表的な正用（落としてよい例）:
- **S-07 受動態**: 既存の定義・既知の事実への言及（「〜と定義される」「〜と呼ばれている」L-24）。
  可能・自発・尊敬の「れる/られる」も受動ではない。
- **S-05 連用形**: 2文に割ると同一主語が連続してしまう場面で、後件を主述文にし主語を変えてある場合
  （A and B 型。例「てんかんは〜神経疾患であり、〜悪影響を及ぼすことが知られている」）。
- **S-10「〜がある」/ S-13「解析・調査」**: 直前文と論理的に繋がり、語順を入れ替えると不自然になる場合。

誤用と判断したものは残し、Tipsの趣旨に沿った**具体的な書き換え案**を添える。

**(b) H系（ハイブリッド）** — 静的に候補が出るが判定は文脈。catalog の H-01〜H-11 を当てる。例:
- H-01「〇〇では」で書き出す文 → その〇〇は読者にとって既知か（未出なら不適切）。
- H-05 主観形容詞（高い/低い/大きい）→ 比較対象が明示されているか。状態変化形「高くなった」の誤用。
- H-09 複文「AはB、CはD」/ H-10「それぞれ」の対応漏れ。

**(c) L系（純粋に文脈依存）** — catalog の L-01〜L-34。特に重要な判定指針:

| 観点 | 見るところ |
|------|-----------|
| L-02 新情報後回し | 各文の主語/冒頭が、直前の文で既出の語から始まっているか（既知→未知の流れ） |
| L-03 トップヘビー | 結論・主題が文/段落の先頭にあるか。理由・条件（〜ため/〜ので）が前置されていないか |
| L-04 トピックセンテンス | 段落の第1文がその段落の主題を述べているか（1段落1テーマ） |
| L-06 具体例後出し | 固有名・略語・具体例より先に、上位概念を置いているか |
| L-08 時間を戻さない | 過去研究→However→本研究の流れ。本研究に移った後で過去知見へ戻っていないか |
| L-09 主語一致 | 「AはB（であり/し）、Cである」でCの主語がAと整合するか |
| L-11 粒度統一 | 並列（A and B / A,B,C / not only〜but also）の各要素が句↔節・質的に揃っているか |
| L-16 係り受け距離 | 主語と述語の間に長い修飾語が挟まっていないか |
| L-17 概要 | アブストがイントロの要約になっていないか（全体の要約か。背景が半分を占めていないか） |
| L-18 序論構成 | 大→小・過去→未来・直線ロジック・4段落（話題→技術課題→既存と不足→本研究） |
| L-19/20 事実と意見 | 事実・他人の知見・自分の意見が分かれているか。序論の自説は問題設定/仮説のみか |
| L-21 課題設定 | 「向上の余地」「実用化に至らず」等の曖昧な課題でなく、具体的な未解決問題か |
| L-27/28 結果の図表 | 図を見なくても結果が伝わるか（図の種類・数値の対応・メッセージ）。「図に示す」の羅列でないか |
| L-30 母集団 | 平均値などに対し標本集合（母集団）が明示されているか |
| L-31/32 考察 | 冒頭に結果要約→仮説と照合・文献参照。先行研究との単なる「答え合わせ」になっていないか |

章ごとのルール（L-17〜L-34）は、対象がその章を含むときだけ適用する。

### Step 3. 統合レポート

重大度ごとにまとめ、各指摘は次の形式で出す。場所が特定できるものは必ず `行:列` を付ける。

```
## 添削レポート — <file>（<文書種別>）

### ✗ error（要修正）
- `path:line:col` [S-01] 「近年」…  → 修正案: …

### △ warning（要検討）
- `path:line:col` [S-07] 受動態…  → （正用と判断したものは除外済み。残: …）

### 💡 文脈（H/L）
- `path:line` [L-03] トップヘビー: この文は理由が先行。「<結論>。これは<理由>のためである」へ
- `path:line` [L-09] 主語一致: 「AはBであり、Cである」のCの主語が…

### まとめ
2〜4行で、最も効く改善（多くは構成・論理）を優先順に。
```

---

## 方針（重要）

- **狼少年にしない**: 件数を競わない。確信度の高い指摘を、Tipsの理由とともに、具体的な書き換え案つきで示す。
  S-soft の機械的 warning を鵜呑みにして大量に並べない。文脈で正用なら落とす。
- **理由を添える**: 「なぜそれが問題か」をTipsの趣旨で短く説明する。ルールIDを併記し、詳細は catalog 参照。
- **書き換えを示す**: Tips は必ず ○ の改善例を持つ。指摘だけで突き放さず、改善案を出す。
- **優先順位**: 文章の構成・論理（L系：トップヘビー/トピックセンテンス/序論構成/主語一致）は、語句の
  言い換え（S系）より影響が大きい。レポートのまとめでは構成・論理を先に挙げる。
- **長文**: 章/節が多い長文は、章ごとに区切って通読・指摘すると漏れが減る。
